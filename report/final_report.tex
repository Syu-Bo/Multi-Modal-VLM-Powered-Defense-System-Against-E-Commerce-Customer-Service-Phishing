\begin{abstract}
傳統釣魚偵測依賴網域黑名單或純文字語意分析，對圖片化訊息、藏惡意 QR Code 的截圖、
與像素級仿冒官方 UI 的假登入頁存在防禦盲區。本專案提出一套開源、可在地化部署的
雙軌制多模態防禦管道（Dual-Track Multi-Modal Pipeline）：以視覺語言模型（VLM）
辨識品牌仿冒與 UI 異常，以大型語言模型（LLM）剖析文字社交工程意圖，再經融合模組產出
結構化威脅報告。本文涵蓋系統設計、資安評估結果，以及架構限制與改進方向。
\end{abstract}

\section{系統設計 (System Design)}
%==========================================================

\subsection{整體架構與資料流}
系統採雙軌制管道，讓視覺與文字模型於獨立支線各自輸出結構化風險特徵，再由融合模組以
加權方式產生最終威脅指數。兩支線可並行推論、支援降級運作，且風險分數可解釋。

\begin{verbatim}
使用者輸入 URL
      |
  fetch_page()  抓取 HTML / 可見文字 / 圖片 / 註冊網域
      |
      +--> Vision Track (VLM)  --> vision_features (risk_score, ...)
      |
      +--> Text  Track (LLM)  --> text_features  (risk_score, ...)
      |
   fuse()  加權融合 (W_v, W_t) + 否決規則
      |
   FusedVerdict  結構化威脅報告 (危險等級 / 緩解建議)
\end{verbatim}

\subsection{結構化輸出 Schema}
兩支線統一輸出可被融合模組穩定解析的 JSON（\texttt{TextVerdict} / \texttt{VisualVerdict}）：
\begin{lstlisting}
{
  "risk_score": 0.0,          // [0,1]
  "confidence": 0.0,          // [0,1]
  "reasons": [ ... ],         // 自然語言推理，供 error analysis
  "suspicious_phrases": [ ... ],
  "detected_language": null
}
\end{lstlisting}

\subsection{融合決策邏輯}
採加權函式加否決規則：
\begin{equation}
ThreatScore = W_v \cdot Score_{vision} + W_t \cdot Score_{text}, \qquad W_v + W_t = 1
\end{equation}
初始權重 $W_v=0.55,\ W_t=0.45$，反映視覺通道為現代電商釣魚主要載體之假設；
最終權重以驗證集 Grid Search 決定。風險分數 $\ge$ 0.7（\texttt{ALERT\_THRESHOLD}）判定為釣魚。
頁面若有多張截圖，頁面視覺分數取該頁所有圖之 max。

\subsection{QR Code 子分析（視覺軌）}
針對提案核心案例一（蝦皮假客服 + QR Code 釣魚），純文字 LLM 無法讀取藏於圖片內 QR Code
的網址。視覺軌加入 \texttt{qr\_decode.py}：以 OpenCV \texttt{QRCodeDetector}（或 pyzbar）解碼頁面
圖片中的 QR Code，比對其網址之註冊網域與頁面網域；若不符（如 QR 指向 \texttt{shoppe88.com}
而頁面宣稱 \texttt{shopee.tw}）即視為高信度釣魚訊號（\texttt{risk\_score}=0.9），作為融合模組之否決
依據。實測：仿冒 QR 評為 0.90、相符 QR 評為 0.30。

\subsection{提示工程管道}
依 ALP 文獻採三層式 Prompt：(1) System Prompt 定義角色與輸出格式；(2) Few-shot 校準本土
話術；(3) User Prompt 注入待測內容。文字/URL 支線另設 Domain-Brand Mismatch 規則：
若頁面提及之品牌與註冊網域不符，強制提高風險分數並標記 \texttt{[DOMAIN\_MISMATCH]}。
此分層亦呼應 OWASP LLM01（Prompt Injection）之防禦建議。

\subsection{LoRA 微調（質化觀察）}
實作端已於 Unsloth Studio 對文字分支進行 QLoRA 微調（\texttt{unsloth/llama-3-8b-bnb-4bit}，
4-bit、LoRA rank 16、context 2048、lr 2e-4，30 steps）。訓練 loss 由約 2.1 收斂至 1.6，eval loss
收斂至約 1.5，顯示模型已學會輸出格式與任務結構、且於驗證集無明顯過擬合。

然而推論測試出現三類問題：(1) 產生與 \texttt{risk\_score} 不一致的 reason；(2) 欄位名稱拼寫錯誤；
(3) 過度引用訓練資料的固定敘述（如「URL was labeled good in training data」）。此即典型的
Dataset Shortcut：在資料量偏少、且標註含啟發式捷徑訊號時，模型傾向記憶捷徑而非學習語意。
改進方向為擴充 Hard Samples、重構 reason 欄位內容，並以 Teacher VLM 重新標註。

URL 分支已備妥 1:1 平衡的 \texttt{url\_train.jsonl}（540 筆），其 gold 理由正好教導 URL 啟發式特徵，
為後續以 LoRA 進一步提升 URL 分支 Recall 之自然延伸（列為未來工作；須注意同一捷徑風險）。

\section{架構限制與改進方向 (Architectural Limitations)}
%==========================================================

\begin{enumerate}[leftmargin=1.4em]
\item \textbf{模型容量限制（已實證）。} URL 分支於 \texttt{llama3.2:3b} 僅 F1 0.316、Recall 0.20；
誤判分析顯示 3b 只能 pattern-match 教科書級釣魚，無法對可疑 TLD、裸 IP、短網址、品牌
token 不符等訊號進行推理。$\rightarrow$ 改進：改用 8b 與針對 URL 特徵的 LoRA 微調。

\item \textbf{視覺資料為弱標註（已實證）。} 圖片僅依是否來自釣魚頁標註，不檢視圖片本身是否
具視覺釣魚特徵；裝飾圖、商品圖亦被標為釣魚。於 143 頁對齊集中，文字與視覺 gold 有
20 頁標籤不一致。$\rightarrow$ 改進：以 Teacher VLM 重新標註、引入 Hard Sample Mining。

\item \textbf{資料不平衡。} 文字 train 釣魚與正常比約 1:3.6，使模型偏向判正常、壓低 Recall。
$\rightarrow$ 改進：對 LoRA 訓練集做平衡採樣（URL 分支已採 1:1）。

\item \textbf{資料洩漏與標註雜訊（已修正）。} 稽核發現原始文字 train/valid 有 10 個網域跨子集
（資料洩漏，使指標虛高）、7 筆重複樣本，以及灰區錯標（如 \texttt{barclays-banking.net}
仿冒銀行卻標為正常）。已以網域分組重切（零洩漏）、去重、修正明確錯標，並保留 25 筆灰區
樣本待人工複核。$\rightarrow$ 提醒：洩漏修正後指標通常會下降，屬更誠實之估計。

\item \textbf{LoRA 之 Dataset Shortcut（已觀察）。} 微調後模型出現「URL was labeled good in
training data」等過度引用訓練敘述、reason 與 risk\_score 不一致、欄位拼寫錯誤等現象，
顯示資料量偏少且標註含捷徑訊號。$\rightarrow$ 改進：擴充 Hard Samples、重構 reason 欄位。

\item \textbf{驗證集規模偏小。} 文字 valid 僅 39 筆（10 個正樣本），Recall 每變動一筆即跳 0.1，
指標雜訊大、結論統計上不穩固。$\rightarrow$ 改進：擴充 valid 正樣本至 30 筆以上。

\item \textbf{推論延遲過高。} 文字分支 P95 約 68--69 秒（Ollama / CPU 推論），遠超 $\le$3000ms
之即時部署目標。$\rightarrow$ 改進：GPU 部署、4-bit 量化、批次化推論。

\item \textbf{URL gold 為啟發式自動標註。} 非人工校驗，邊界樣本雜訊大。$\rightarrow$ 改進：人工抽驗校正。
\end{enumerate}

\section{自動化處置 Playbook (Incident Response)}
%==========================================================
當系統判定高風險釣魚時，\texttt{playbook\_generator.py} 依威脅判定（URL／品牌／風險分數）
自動產生可執行的處置文件，分三部分：
\begin{enumerate}[leftmargin=1.4em]
\item \textbf{網域調查指令}：針對註冊網域產生 \texttt{whois} / \texttt{dig} / \texttt{curl -I} 指令，
並附 VirusTotal、urlscan.io 查詢連結與 registrar abuse 下架管道（確定性模板）。
\item \textbf{165 檢舉信草稿}：以 few-shot 引導 LLM 產生條列式繁中檢舉信。
\item \textbf{使用者防詐宣導訊息}：產生語氣親切、明確之警示，導引至官方查證與 165 專線。
\end{enumerate}
LLM 部分可切換模型，端點不可用時自動降級為靜態模板。此模組對應 term project 之
「Incident Response \& Playbook Generation」範疇；範例輸出見 \texttt{playbook/example\_shoppe88.md}。
