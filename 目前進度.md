# 基於多模態視覺語言模型之電商與假客服釣魚防禦系統：評估與 LoRA 訓練說明

## 1. 作業目標

本作業主要目標是建立一套釣魚網站偵測流程，並完成以下四項工作：

1. 建立文字分支資料集，包含釣魚網站與正常網站樣本。
2. 使用 Llama 系列模型進行釣魚網站文字內容判斷。
3. 設計多版本 Prompt，並比較不同 Prompt 對模型表現的影響。
4. 使用 Unsloth Studio 進行 LoRA / QLoRA 微調，觀察模型在結構化 JSON 輸出與釣魚網站判斷任務上的改善情形。

本次作業的實驗重點集中於 Text Branch，也就是根據網頁 URL、Page Title 與 Visible Page Text 進行風險判斷。Vision Branch / VLM 模型目前仍在訓練與驗證中，因此尚未納入本次結果。

## 2. 資料集設計

### 2.1 資料來源

本研究使用的資料來源包含：

- 釣魚網站 URL feed
- `phishing_site_urls.csv` 公開 URL 標籤資料
- 可成功抓取內容的正常網站頁面
- 可成功抓取內容的釣魚網站頁面

其中，`bad` 標籤作為 phishing samples，`good` 標籤作為 legitimate samples。

### 2.2 文字資料格式

文字資料轉換為 ChatML / JSONL 格式，每筆資料包含三個角色：

```json
{
  "messages": [
    {
      "role": "system",
      "content": "You are a defensive cybersecurity analyst specialising in phishing detection..."
    },
    {
      "role": "user",
      "content": "Page URL: ...\nPage title: ...\n<PAGE>... </PAGE>"
    },
    {
      "role": "assistant",
      "content": "{\"risk_score\":0.05,\"confidence\":0.75,\"reasons\":[...],\"suspicious_phrases\":[],\"detected_language\":null}"
    }
  ]
}
```

其中：

- `system`：定義模型角色、任務與輸出格式。
- `user`：提供網頁 URL、標題與可見文字內容。
- `assistant`：提供標準答案，格式為 JSON。

### 2.3 資料切分

本次文字分支資料共建立 197 筆樣本：

| Dataset | 筆數 | 用途 |
|---|---:|---|
| Train | 158 | LoRA / QLoRA 訓練 |
| Validation | 39 | Prompt 評估、模型效能驗證 |

訓練集僅用於 LoRA 微調，不用於 Prompt Iteration 或 Fusion Grid Search，以避免資料洩漏。


### 2.4 視覺資料 Parquet 格式

除了文字分支的 JSONL 資料外，本研究也已完成圖片分支資料的前處理，並將視覺訓練資料轉換為 Parquet 格式，以利後續匯入 Unsloth Studio 進行 VLM / Vision Branch 微調。由於視覺語言模型訓練需要同時包含圖片內容、文字 prompt 與標準答案，單純 JSONL 格式不適合直接保存圖片物件，因此本研究將圖片以 bytes 形式寫入 Parquet 欄位中。

目前圖片分支資料包含：

| 檔案 | 用途 |
|---|---|
| `visual_train_studio.parquet` | Vision Branch / VLM 訓練資料 |
| `visual_valid_studio.parquet` | Vision Branch / VLM 驗證資料 |

Parquet 檔案主要欄位如下：

| 欄位名稱 | 說明 |
|---|---|
| `image` | 儲存圖片資料，通常包含 `bytes` 與 `path`。`bytes` 為圖片二進位內容，`path` 為圖片檔名或來源名稱。 |
| `prompt` | 提供給 VLM 的文字提示詞，通常包含要求模型判斷圖片是否具有釣魚網站特徵的任務說明。 |
| `answer` | 模型訓練時的標準答案，通常為 JSON 格式，包含風險分數、理由與判斷結果。 |
| `messages` | 將 prompt、image placeholder 與 assistant answer 組合成 Unsloth / OpenAI conversation 格式，方便平台建立圖文對話訓練資料。 |
| `source_image_path` | 原始圖片所在路徑，用於追蹤資料來源、除錯與後續人工檢查。 |

此資料處理流程代表 Vision Branch 目前已進入「可訓練資料準備完成」階段。後續將如同文字分支一樣，將 `visual_train_studio.parquet` 與 `visual_valid_studio.parquet` 匯入 Unsloth Studio，進行 VLM fine-tuning、模型推論與 Precision、Recall、F1-score 等指標評估。由於 VLM 訓練與驗證仍在進行中，本次報告尚未納入圖片分支的正式評估結果。

## 3. 評估指標設計

本研究採用 `risk_score >= 0.7` 作為 phishing 判定門檻：

```text
risk_score >= 0.7 → phishing
risk_score < 0.7  → legitimate
```

根據模型預測結果與標準答案，可得到 TP、FP、TN、FN，並進一步計算以下指標。

### 3.1 Precision

Precision 表示模型判定為釣魚網站的樣本中，有多少比例實際上為釣魚網站。

```text
Precision = TP / (TP + FP)
```

Precision 越高，代表誤判正常網站為釣魚網站的情況越少。

### 3.2 Recall

Recall 表示所有真正的釣魚網站中，模型成功偵測出的比例。

```text
Recall = TP / (TP + FN)
```

Recall 越高，代表漏判釣魚網站的情況越少。

### 3.3 F1-score

F1-score 綜合 Precision 與 Recall，用於衡量整體分類表現。

```text
F1 = 2 × Precision × Recall / (Precision + Recall)
```

### 3.4 False Positive Rate, FPR

FPR 表示正常網站被誤判為釣魚網站的比例。

```text
FPR = FP / (FP + TN)
```

### 3.5 JSON Schema Compliance

JSON Schema Compliance 用於檢查模型是否能輸出符合格式要求的 JSON。

```text
JSON Schema Compliance = 成功解析 JSON 的筆數 / 總筆數
```

### 3.6 P95 Latency

P95 Latency 表示所有推論時間中第 95 百分位的延遲，用於衡量系統部署時的即時性。

## 4. 評估程式流程

### 4.1 評估流程說明

1. 讀取 `datasets/text_valid.jsonl`。

## 5. Prompt 版本設計

### 5.1 Prompt v1：Baseline

Prompt v1 為基礎版本，只要求模型根據網頁內容輸出 JSON 結果，未加入額外規則。

此版本主要作為比較基準，用於觀察模型在沒有額外提示策略下的原始表現。

### 5.2 Prompt v2：Domain-Brand Mismatch Rule

Prompt v2 在 v1 基礎上加入 Domain-Brand Mismatch 規則。

核心概念是：如果網頁內容提到知名品牌，但實際網域並非官方網域，則視為高風險釣魚訊號。

範例：

```text
- Text mentions Shopee / 蝦皮, but domain is not shopee.tw or shopee.com.
- Text mentions momo, but domain is not momoshop.com.tw.
- Text mentions LINE Pay, but domain is not line.me.
```

此版本在本次實驗中取得最佳 F1-score，因此後續推論階段採用 Prompt v2 作為主要提示詞基礎。

### 5.3 Prompt v3：v2 + Taiwan Few-shot

Prompt v3 在 v2 基礎上加入台灣常見詐騙案例 Few-shot，例如：

- 蝦皮帳號異常驗證
- momo 分期付款失敗
- 黑貓宅急便包裹地址錯誤

然而本次實驗結果顯示，Prompt v3 並未進一步提升效能，推測原因是 Few-shot 範例數量有限，且與驗證資料分布不完全一致，造成模型過度依賴特定模式。

## 6. Prompt 評估結果整理

| Prompt | TP | FP | TN | FN | Precision | Recall | F1-score | FPR |
|---|---:|---:|---:|---:|---:|---:|---:|---:|
| v1 Baseline | 3 | 3 | 26 | 7 | 0.50 | 0.30 | 0.3750 | 0.1034 |

## 9. LoRA 訓練結果

### 9.1 訓練概要

本次 QLoRA 訓練共執行 30 steps，最終 Training Loss 約為 1.6081，耗時約 6 分鐘，處理約 261,123 個 tokens。

本次訓練整合觀察四項指標：

1. Training Loss
2. Gradient Norm
3. Learning Rate
4. Eval Loss

- Training Loss 由約 2.3 下降至約 1.6，表示模型逐步學習資料集中之任務格式與風險判斷模式。

整體而言，四項指標皆呈現穩定且合理的訓練趨勢，代表本次 QLoRA 微調流程順利完成。

### 11.2 模型輸出觀察

模型已能輸出符合 JSON 結構的結果，但仍出現以下問題：

- `risk_score` 與 `reasons` 內容可能不一致。
- 模型可能引用訓練資料中的固定語句，例如 `URL was labeled good in training data`。
- 模型可能學到資料集標籤捷徑，而不是完全依據頁面內容與釣魚特徵判斷。

### 11.3 問題原因分析

此現象可能與訓練資料中的 reasons 欄位設計有關。例如部分 legitimate 樣本的標準答案包含：

```text
URL is labeled good in the source dataset and the page was reachable.
```

此類描述容易讓模型學到「資料集標籤捷徑」，而非實際根據頁面文字與 URL 特徵進行推論。

### 11.4 改善方向

後續可採取以下方式改善：

1. 移除 assistant 標準答案中與 source label 相關的描述。
2. 將 reasons 改寫為可觀察的網頁風險特徵。
3. 增加 Shopee、momo、LINE Pay 等台灣常見品牌的 hard samples。
4. 加入更多 domain-brand mismatch 案例。
5. 使用 error cases 進行 Hard Sample Mining。

## 13. 結論

本研究完成基於大型語言模型之釣魚網站偵測流程，包含資料集建立、Prompt Engineering、模型評估與 LoRA 微調實驗。Prompt 評估結果顯示，加入 Domain-Brand Mismatch 規則的 Prompt v2 取得最佳 F1-score，表示品牌名稱與網域一致性判斷對釣魚網站偵測具有實際幫助。

在 LoRA 微調方面，模型已能學習 JSON 輸出格式與部分風險判斷模式，且訓練過程中 Training Loss 與 Eval Loss 皆呈現穩定下降。然而，實際測試仍發現模型可能產生推論邏輯不一致與資料集捷徑學習現象，顯示後續仍需擴充資料量並改善標註內容品質。

整體而言，本次作業完成 Text Branch 釣魚網站偵測模型之初步建置與評估，後續將進一步完成 VLM 分支訓練與多模態 Fusion 實驗，以提升系統於真實釣魚網站場景中的辨識能力與泛化表現。
